# Phase 7 Explained

## What we built

We turned a three-hour passive Kalshi Level-2 recording into a repeatable research pipeline. The source capture stays unchanged. The pipeline creates a cleaned event history, a sequence of market snapshots/features, and two controlled backtest outputs.

## How the data moves

```text
raw WebSocket frames -> normalized observed events -> observed book cursor -> causal features -> configured backtest -> result manifest/orders/fills
```

The raw file retains the exact incoming WebSocket text and local receive time. Normalization checks message structure, ticker consistency, source sequences, quantities, duplicates, and ordering. It writes a new canonical event stream with hashes that point back to the raw capture.

The observed-book cursor applies the initial snapshot and each later delta in recorded order. It exposes a stable watermark: a strategy may only use the book state that has already been applied at that watermark. Features such as bid, ask, spread, midpoint, depth, imbalance, and last trade are generated from that state only, which prevents future-data leakage.

## Why observed data is outside the exchange

The historical book tells us what Kalshi published. It does not tell us every original order, queue position, hidden order, or whether our hypothetical order would have executed. Replaying it through `ExchangeSimulator` would incorrectly make a reconstruction look like observed exchange truth. The C++ exchange therefore remains unchanged as the authoritative simulator of its own synthetic commands; the historical pipeline is a separate observed-market projection.

## What the two backtests mean

`no_fill_v1` is the control: it creates passive, risk-admitted simulated quotes but never fills them. `trade_touch_v1` allows a public trade at or through a quote price to consume simulated quantity in deterministic order-ID order. Both use explicit 100 ms market-data, decision, and order latency assumptions, one-contract quotes, 30-second logical expiry, and a 10-contract inventory limit.

The touch model produced 76 simulated fills; the no-fill control produced none. This demonstrates that the feature, scheduling, risk, inventory, and artifact paths work together. It does not demonstrate that a live trader would have received 76 fills or earned a profit.

## Why the manifests matter

Every generated stage records hashes of its inputs and outputs, its schema/version, source fidelity, truth category, assumptions, and limitations. Re-running the V1 backtest with the same configuration generated byte-identical result files. This makes later comparisons inspectable and helps prevent accidentally comparing results made from different data or assumptions.

## Where we stopped deliberately

We did not add fees, PnL, settlement, collateral, calibrated queue/fill behaviour, paper trading, or live execution. Those need separate accounting and execution-realism designs. The next phase should make those assumptions explicit, test them with sensitivity analysis, and continue to label non-observed outputs honestly.
